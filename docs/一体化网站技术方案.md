# 一体化网站技术方案（个人博客 + AI工具 + 软件资源 + 影视）

## 1. 目标与约束

### 1.1 当前资源
- 服务器：4核 CPU / 4GB RAM / 3Mbps 带宽
- 运维面板：1Panel
- 目标：一个统一站点，集成
  - 个人博客
  - AI 工具服务
  - 软件资源导航/下载页
  - 影视内容浏览与播放入口

### 1.2 关键约束（必须正视）
- 3Mbps 带宽非常紧张，不适合自建高并发视频流媒体。
- 4GB 内存下，不适合同时运行过多重型组件（例如本地大模型推理、复杂搜索集群、转码服务）。
- 影视与软件资源需要合法合规，避免侵权分发与高风险内容托管。

### 1.3 设计原则
- KISS：优先可上线、可维护的最小方案。
- YAGNI：首期不做本地大模型、不做复杂推荐系统。
- DRY：统一账号、统一标签体系、统一内容管理后台。
- SOLID：模块边界清晰，博客/AI/资源/影视可独立演进。

## 2. 功能改造建议（重点）

## 2.1 信息架构（统一门户）
- `/` 首页：聚合入口（最新博客、热门工具、最新资源、影视推荐）
- `/blog` 博客模块
- `/ai` AI 工具模块
- `/software` 软件资源模块
- `/movies` 影视模块（合规聚合）
- `/admin` 管理后台

## 2.2 模块调整建议

### A. 博客模块（核心内容引擎）
- 保留 Markdown 写作体验。
- 增加：标签、分类、全文检索、草稿/发布状态、SEO 元信息。
- 增加：文章关联 AI 工具和软件资源（提升内部导流）。

### B. AI 工具模块（建议做“服务编排”，不做重推理）
- 建议模式：调用第三方大模型 API（OpenAI/通义/智谱等），你自己做工具编排层。
- 功能建议：
  - Prompt 模板库（可复用）
  - 调用配额与限流（按用户/IP）
  - 调用日志与成本统计（每日费用可视化）
  - 异步任务机制（长任务排队）
- 不建议首期做本地 GPU 推理（当前机器不匹配）。

### C. 软件资源模块（建议做“索引+评测+合法下载指引”）
- 资源实体：名称、版本、平台、官网、镜像、哈希、截图、评测文章。
- 强化：
  - “官方下载优先”标识
  - 版本变更记录
  - 用户评分与评论（可后置）
- 安全建议：对上传文件做白名单、大小限制、病毒扫描流程（可异步）。

### D. 影视模块（建议做“合法聚合”，不做自建片库）
- 推荐实现：
  - 仅维护影片信息、评分、海报、预告、播放来源入口
  - 播放优先跳转合法平台或内嵌合法公开源
- 不建议：自托管大量影视文件、实时转码、P2P 分发。
- 原因：3Mbps 带宽和 4GB 内存无法支撑稳定体验，也有较高合规风险。

## 3. 技术架构

## 3.1 推荐技术栈
- 前端：React + Vite + TypeScript + React Router + Zustand（或 Redux Toolkit）
- 后端：FastAPI + SQLAlchemy + Pydantic
- 数据库：MySQL 8（1Panel 一键部署）
- 缓存/队列：Redis（缓存、限流计数、轻量任务队列）
- 反向代理：Nginx（1Panel 网站/反代）
- 对象存储：优先外部对象存储（OSS/COS/S3）；本机仅存小文件
- 前端发布：`npm run build` 生成 `dist/`，由 Nginx 静态托管（`vite preview` 仅用于本地预览）

## 3.2 逻辑分层
- `web`：前端展示与后台管理 UI
- `api`：REST API（认证、内容、搜索、统计）
- `service`：博客服务、AI 编排服务、资源服务、影视聚合服务
- `infra`：MySQL、Redis、对象存储、日志监控

## 3.3 部署拓扑（单机）
- Nginx（80/443，TLS，静态资源缓存）
- React 静态站点容器（Nginx 托管 `dist`）
- FastAPI 容器（`fastapi run ... --proxy-headers --workers 2`）
- MySQL 容器
- Redis 容器

说明：
- FastAPI 官方文档建议容器内可使用多 worker；考虑 4C4G 资源，先从 `workers=2` 起步压测后再调优。
- 反向代理场景建议开启 `--proxy-headers`，并结合 `--forwarded-allow-ips` 配置可信代理来源。

## 4. 数据模型（MVP）

## 4.1 核心表
- `users`：用户信息、角色、状态
- `posts`：博客正文、摘要、状态、SEO 字段
- `tags` / `post_tags`
- `ai_tools`：工具定义、模板、可见性
- `ai_call_logs`：调用日志、耗时、token/费用、状态
- `software_items`：软件基本信息、版本、平台、下载链接
- `software_versions`：版本历史与校验值
- `movie_items`：影片元数据、来源信息、上映时间、地区
- `movie_sources`：播放来源（合法平台 URL）

## 4.2 统一能力
- 所有内容表带 `created_at`、`updated_at`、`is_deleted`。
- 统一标签体系（博客/软件/影视可共用标签）。

## 5. API 设计（示例）

## 5.1 博客
- `GET /api/v1/posts`
- `GET /api/v1/posts/{id}`
- `POST /api/v1/admin/posts`
- `PUT /api/v1/admin/posts/{id}`

## 5.2 AI 工具
- `GET /api/v1/ai/tools`
- `POST /api/v1/ai/run`
- `GET /api/v1/ai/logs`（管理员）

## 5.3 软件资源
- `GET /api/v1/software`
- `GET /api/v1/software/{id}`
- `POST /api/v1/admin/software`
- `POST /api/v1/admin/software/{id}/versions`

## 5.4 影视
- `GET /api/v1/movies`
- `GET /api/v1/movies/{id}`
- `GET /api/v1/movies/{id}/sources`

## 6. 性能与容量规划

## 6.1 带宽预算（3Mbps）
- 峰值可用吞吐约 0.375MB/s。
- 结论：必须把静态资源放 CDN，视频走外部平台，不在本机大流量回源。

## 6.2 资源预算建议
- MySQL：~800MB 上限（含缓冲）
- Redis：~256MB 上限
- FastAPI：2 workers，单 worker 300~500MB 监控后调优
- 预留 1GB 给系统和 Nginx

## 6.3 缓存策略
- 首页聚合接口：30~60 秒缓存
- 热门内容列表：5~10 分钟缓存
- 详情页支持条件缓存（ETag/Last-Modified）

## 7. 安全与合规

## 7.1 安全基线
- 强制 HTTPS（1Panel 证书自动续签）
- JWT + Refresh Token
- 管理后台启用 RBAC
- 输入校验、SQL 注入防护、上传白名单
- 登录与 AI 调用接口限流（Redis）

## 7.2 合规建议
- 影视仅做合法来源聚合，不直接分发侵权内容。
- 软件资源优先官方源，保留来源声明与版权声明。
- 落地隐私政策、免责声明、版权投诉通道。

## 8. 1Panel 配置与运维步骤（React + FastAPI）

## 8.1 基础准备
- 域名解析到服务器公网 IP
- 1Panel 创建网站与反向代理
- 申请并绑定 HTTPS 证书
- 在 1Panel 应用商店安装：MySQL、Redis、OpenResty/Nginx、Docker 运行环境

## 8.2 容器与服务
- 部署 FastAPI 容器（建议 `fastapi run app/main.py --proxy-headers --workers 2 --port 8000`）
- 构建 React 前端：`npm ci && npm run build`
- 将 `dist/` 发布到 1Panel 站点目录（或独立 Nginx 静态容器）
- Nginx 路由规则：
  - `/api` 反代到 FastAPI 容器 `:8000`
  - `/` 指向 React `dist/` 静态资源
  - SPA 刷新兜底：`try_files $uri /index.html;`

## 8.3 环境变量示例
- `APP_ENV=prod`
- `DATABASE_URL=mysql+pymysql://user:pass@mysql:3306/blog`
- `REDIS_URL=redis://redis:6379/0`
- `JWT_SECRET=...`
- `OPENAI_API_KEY=...`（或其它模型服务商）
- `VITE_API_BASE_URL=https://your-domain.com/api/v1`

## 8.4 1Panel 运维基线（建议）
- 日志：开启 API 访问日志、错误日志和 Nginx 访问日志分离保存。
- 监控：在 1Panel 打开 CPU、内存、磁盘、容器重启次数告警。
- 备份：MySQL 每日增量 + 每周全量；站点配置与证书每周备份。
- 发布：采用“蓝绿目录”或“版本目录”发布前端静态资源，保留最近 2 个可回滚版本。
- 安全：
  - 仅开放 `80/443`（数据库端口仅内网可达）
  - 后台路径加访问控制（IP 白名单或二次认证）
  - 定期更新基础镜像与依赖（至少每月一次）

## 9. 迭代计划（6 周）

## 9.1 第 1-2 周（MVP）
- 用户认证
- 博客发布与展示
- 软件资源列表与详情
- AI 工具调用链路（1-2 个工具）

## 9.2 第 3-4 周（增强）
- 影视模块（元数据 + 合法播放来源）
- 后台内容管理
- 缓存与限流

## 9.3 第 5-6 周（稳定）
- 监控告警（CPU、内存、QPS、错误率）
- 日志归档与备份
- 安全扫描与性能压测

## 10. 验收标准（上线门槛）
- 功能：四大模块可访问，后台可管理
- 性能：100 并发下首页 P95 < 1.5s（命中缓存）
- 安全：TLS、鉴权、限流、日志审计到位
- 可运维：可一键重启、可回滚、可备份恢复

## 11. 关键决策摘要
- 不做本地大模型推理，改为 API 编排，控制成本和稳定性。
- 不做自建影视流媒体，改为合法来源聚合，规避带宽与合规风险。
- 以 FastAPI + MySQL + Redis + Nginx 的轻量组合满足首期上线。
- 利用 1Panel 简化部署、证书、容器生命周期管理。
