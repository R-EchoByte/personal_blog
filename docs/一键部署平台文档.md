# 一键部署平台文档

## 1. 文档用途

这份文档只解决一件事：  
**把源码上传到 Ubuntu 服务器后，用最少步骤完成打包、启动、托管和更新。**

适用场景：

- 首次部署到新服务器
- 已部署项目的日常更新
- 排查服务是否正常运行

---

## 2. 部署方式

项目统一采用下面这条路径：

1. 将完整源码上传到 Ubuntu 服务器
2. 在服务器源码目录执行 `make package`
3. 由服务器本机构建前端、校验后端、执行 PyInstaller 打包
4. 使用 `systemd` 托管服务
5. 使用 `nginx` 暴露公网访问

打包后的主程序：

```text
backend/dist/personal_blog
```

默认服务端口：

```text
7869
```

---

## 3. 服务器要求

推荐环境：

- Ubuntu 22.04+
- Node.js 20+
- Python 3.11+
- `uv`
- `make`
- `nginx`

---

## 4. 首次部署

### 4.1 上传源码

建议部署目录：

```bash
/var/www/personal_blog
```

后续命令默认都在这个目录执行。

### 4.2 安装基础依赖

```bash
sudo apt update
sudo apt install -y build-essential make curl nginx
```

### 4.3 安装 Node.js

要求版本不低于 20。  
安装完成后确认：

```bash
node -v
npm -v
```

### 4.4 安装 uv

```bash
curl -LsSf https://astral.sh/uv/install.sh | sh
```

安装后重新登录终端，再确认：

```bash
uv --version
```

### 4.5 执行打包

```bash
cd /var/www/personal_blog
make preflight
make package
```

这一步会自动完成：

1. 前端构建 `frontend/dist`
2. 后端依赖锁定 `uv lock`
3. 后端开发依赖同步 `uv sync --group dev`
4. `ruff check`
5. `ty check`
6. `PyInstaller` 打包

### 4.6 冒烟测试

```bash
make smoke
```

查看版本：

```bash
make show-version
```

### 4.7 安装 systemd 服务

```bash
make install-systemd
make install-watchdog
make restart-service
make status-service
```

确认是否开机自启：

```bash
systemctl is-enabled personal-blog
systemctl is-active personal-blog
```

正常结果应为：

- `enabled`
- `active`

查看健康巡检定时器：

```bash
make status-watchdog
```

说明：

- 巡检定时器会按分钟请求 `http://127.0.0.1:7869/api/v1/health`
- 如果健康检查失败，会自动执行 `systemctl restart personal-blog`

### 4.8 安装 nginx 反向代理

```bash
make install-nginx
```

### 4.9 放行端口

如果启用了 `ufw`：

```bash
sudo ufw allow 80/tcp
sudo ufw allow 7869/tcp
sudo ufw reload
```

如果是云服务器，还要在安全组放行：

- `80/tcp`
- `7869/tcp`

### 4.10 最终验证

先验证应用本机访问：

```bash
curl http://127.0.0.1:7869/
curl http://127.0.0.1:7869/api/v1/health
```

再验证 nginx 转发：

```bash
curl http://127.0.0.1/
curl http://127.0.0.1/api/v1/health
```

公网访问地址：

- `http://<server-ip>/`
- `http://<server-ip>/api/v1/health`

如果你只是临时直连应用端口，也可以访问：

- `http://<server-ip>:7869/`
- `http://<server-ip>:7869/api/v1/health`

## 5. 后续更新

代码更新后，标准流程固定为：

```bash
cd /var/www/personal_blog
git pull
make package
make smoke
make show-version
make restart-service
make status-service
```

如果 `systemd` 或 `nginx` 模板有调整，再补执行：

```bash
make install-systemd
make install-nginx
```

```bash
make preflight

make smoke
make show-version
make install-systemd
make install-watchdog

make restart-service
```

说明：

- `make deploy`：执行打包、冒烟、安装模板、重启服务

## 7. 常见问题

### 7.1 `make smoke` 失败

先看日志：

```bash
cat /tmp/personal_blog.log
```

再确认端口是否被占用：

```bash
ss -ltnp | grep 7869
```

### 7.2 本机能访问，公网不能访问

重点排查两层：

1. Ubuntu 防火墙
2. 云服务器安全组

建议先执行：

```bash
curl -v http://127.0.0.1:7869/
curl -v http://<server-ip>:7869/
ufw status
```

### 7.3 服务启动失败

查看状态和日志：

```bash
systemctl status personal-blog --no-pager
journalctl -xeu personal-blog.service --no-pager
```

## 8. 回滚建议

不要直接覆盖旧版本目录。  
建议保留最近两次发布，例如：

```text
/opt/releases/personal_blog/20260506-1450
/opt/releases/personal_blog/20260507-0930
```

线上目录只保留一个当前软链接：

```text
/var/www/personal_blog -> /opt/releases/personal_blog/<current>
```

这样回滚时只需要切回旧目录并重启服务。
