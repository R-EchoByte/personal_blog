# 下一阶段开发规划（React + FastAPI + 1Panel）

## 1. 当前基线（已完成）

- 前后端骨架已搭建：`frontend/`、`backend/`
- 首页首屏已实现：全屏背景、语录卡片、快捷入口
- 背景图机制已实现：`public/bj*` 刷新随机切换
- 背景清单自动同步：`npm run bg:sync` 在 `dev/build` 自动触发

## 2. 规划目标（接下来 3 周）

- 完成“可用 MVP”：博客、AI 工具、软件资源、影视聚合四模块可访问
- 建立统一 API 规范与错误处理规范
- 打通 1Panel 部署链路（前端静态 + 后端容器 + Nginx 反代）
- 保持代码结构简单，不引入过度分层

## 3. 分阶段实施（建议）

## 3.1 第 1 阶段：路由与页面骨架（2-3 天）

目标：先把前端信息架构跑通。

任务：
1. 前端新增页面：
   - `/blog`
   - `/ai`
   - `/software`
   - `/movies`
2. 首页快捷入口改为前端路由跳转（保留外链 GitHub）。
3. 每个页面先做统一“列表 + 占位空状态”。

验收：
1. 路由可访问且无 404。
2. 移动端不溢出、不卡布局。

## 3.2 第 2 阶段：后端 API 最小闭环（4-5 天）

目标：让前端能拿到真实数据。

任务：
1. 新增 API endpoint（先用内存/静态数据）：
   - `GET /api/v1/posts`
   - `GET /api/v1/ai/tools`
   - `GET /api/v1/software`
   - `GET /api/v1/movies`
2. 统一响应结构（成功/失败/错误码）。
3. 前端 `services/api.ts` 增加基础错误处理与超时策略。

验收：
1. 前端四页面都能从 API 拉取列表。
2. 接口错误时前端能显示友好提示。

## 3.3 第 3 阶段：管理与发布能力（4-6 天）

目标：把内容更新流程建立起来。

任务：
1. 后端新增最小管理接口（可先用简单 Token）：
   - 新增/修改文章
   - 新增/修改软件条目
   - 新增/修改影视来源
2. 前端补一个轻量管理页（先不做复杂后台框架）。
3. 增加基础日志与操作审计字段（创建时间、更新时间、操作者）。

验收：
1. 可通过管理接口更新数据并在前台可见。
2. 关键操作有日志可追踪。

## 3.4 第 4 阶段：1Panel 上线与运维（2-3 天）

目标：可稳定上线。

任务：
1. 前端构建并发布静态资源到 Nginx。
2. 后端容器化并由 1Panel 托管。
3. Nginx 配置 `/api` 反代与 HTTPS。
4. 配置基础监控与备份（数据库每日备份）。

验收：
1. 域名可访问首屏与 API。
2. 重启容器后服务能自动恢复。

## 4. 任务优先级（P0/P1/P2）

- P0（必须先做）：
  - 四模块路由与页面骨架
  - 四模块只读 API
  - 1Panel 部署链路
- P1（建议随后做）：
  - 管理接口与管理页
  - 统一错误码与日志
- P2（后续优化）：
  - 搜索、标签联动、推荐位
  - 细粒度权限、操作审计面板

## 5. 代码与文档约束

1. 前端新增页面只放在 `src/pages`，公共组件放 `src/components`。
2. 后端新增业务路由统一注册到 `app/api/router.py`。
3. 所有新接口都补文档到 `docs/`，避免“代码有、文档无”。
4. 优先复用已有样式变量与布局，避免 UI 失控。

## 6. 本周建议执行清单（可直接开始）

1. 完成四模块前端页面与路由占位。
2. 完成四模块后端只读接口。
3. 前端接入四模块接口并展示列表。
4. 做一次本地联调与构建验证：
   - `frontend: npm run build`
   - `backend: python -m compileall app`

## 7. 风险与应对

- 风险：需求扩张导致结构膨胀  
应对：严格按 P0/P1/P2 推进，不做超前功能。

- 风险：3Mbps 带宽影响影视体验  
应对：影视只做合法来源聚合，不做本机流媒体分发。

- 风险：上线后回滚困难  
应对：前端保留最近 2 个构建版本，后端镜像版本化发布。
